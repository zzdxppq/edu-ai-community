# 河南省城乡学校共同体AI智能助手建设方案

---

## 一、项目概述

### 1.1 项目名称

河南省城乡学校共同体智能问答助手（以下简称"校共体AI助手"）

### 1.2 项目背景

根据《河南省教育厅 河南省财政厅 河南省人力资源和社会保障厅关于深入推进城乡学校共同体建设工作的通知》及《河南省县域结对型城乡学校共同体建设指南》，河南省正在深入推进县域城乡义务教育学校共同体建设。

当前校共体建设面临以下痛点：

| 序号 | 痛点 | 现状表现 |
|------|------|---------|
| 1 | 政策文件分散 | 多部门、多层级文件交叉，一线教育工作者难以全面掌握 |
| 2 | 解读门槛高 | 政策语言专业性强，不同角色关注点不同，缺乏个性化解读 |
| 3 | 实践经验难复用 | 优秀案例散落各地，缺乏统一归集、脱敏和推荐机制 |
| 4 | 流程指导缺乏个性化 | 线下培训覆盖面有限，无法针对具体校情提供差异化指引 |
| 5 | 信息传递效率低 | 依赖文件下发和集中培训，响应周期长、触达率低 |

### 1.3 项目定位

本项目定位为集 **"政策解读器、规划引导员、案例推荐官、流程导航仪"** 于一体的智能化业务赋能平台。以对话式交互为核心载体，嵌入校共体网站（edu.51xiaoping.com），面向6类角色提供7×24小时在线智能服务，一次性全功能交付。

## 二、建设目标

### 2.1 总体目标

构建一个嵌入校共体网站的智能问答助手，通过自然语言交互，将静态政策文件与动态实践经验转化为**可交互、可执行、可追溯**的个性化指导服务，有效降低制度性成本，提升校共体建设的规范化、科学化与智能化水平。

### 2.2 具体目标

| 目标维度 | 具体指标 |
|---------|---------|
| **精准服务** | 基于知识库实现政策问答准确率≥85%，每条回答强制引用政策出处 |
| **角色适配** | 面向6类角色（管理者、牵头校、城镇成员校、乡村成员校、研究人员、其他）提供差异化服务 |
| **响应效率** | 95%的查询3秒内响应，峰值支持≥500并发用户 |
| **知识进化** | 建立"用户反馈→运营审核→知识入库"闭环，实现知识库持续自进化 |
| **安全合规** | 全程HTTPS加密、敏感信息脱敏、双向敏感词过滤、AI输出免责声明 |
| **低门槛使用** | 支持口语化提问、错别字容错，乡村教师首次使用即可完成基本问答 |

## 三、服务对象与角色体系

### 3.1 服务对象

本系统面向河南省城乡学校共同体建设的全链条参与者，覆盖6类核心角色：

| 角色 | 描述 | 核心诉求 | 服务侧重 |
|------|------|---------|---------|
| **区域教育管理人员** | 市、县级教育行政部门管理者 | 快速获取政策解读、管理决策支持 | 宏观政策分析、区域对标 |
| **校共体核心牵头校人员** | 校共体牵头学校校长/骨干教师 | 结对指导、经验分享、流程规范 | 结对模式指引、管理经验 |
| **校共体成员校人员（城镇）** | 城镇学校参与校共体的教师 | 教学案例学习、资源获取 | 实践案例、课程资源 |
| **校共体成员校人员（乡村）** | 乡村学校参与校共体的教师 | 低门槛操作、直白易懂的指导 | 简洁指引、操作步骤 |
| **研究人员** | 教育研究者 | 深度政策分析、数据文献引用 | 深度解读、文献溯源 |
| **其他人员** | 关注校共体建设的社会人士 | 了解校共体基本情况 | 通用问答 |

### 3.2 运营管理角色

| 角色 | 职责 |
|------|------|
| **运营人员** | 知识库管理、内容审核、反馈处理、配置维护、数据分析 |

### 4.2 核心功能清单

#### 4.2.1 智能问答引擎

| 功能项 | 说明 |
|--------|------|
| 精准政策问答（RAG） | 基于知识库政策文件，通过混合检索（向量+关键词+重排序）生成精准回答 |
| 强制引用溯源 | 每条政策回答必须标注文件名称、章节条款，引用可点击跳转原文 |
| 多轮对话管理 | 保持对话连续性，支持连续追问，最大30轮/次，上下文Token≤8000 |
| 话题边界感知 | 检测话题跳跃，温和确认用户意图 |
| 语义缓存 | 高频问题缓存命中率目标≥30%，按角色分组，知识库更新自动失效 |
| 流式输出 | SSE逐字流式展示回答，提升体验感知速度 |

#### 4.2.2 知识库与文档处理

| 功能项 | 说明 |
|--------|------|
| 文档解析管道 | 支持PDF、Word格式，自动提取文本、表格、流程图（视觉模型识别） |
| 语义切分 | 按章-条-款结构切分，500-1000 token/片段，50-100 token重叠 |
| 向量化入库 | 片段向量化后存入PostgreSQL + pgvector，支持HNSW索引加速 |
| 混合检索 | 向量语义检索 + 关键词全文检索 + Rerank重排序，取Top-5 |
| 角色可见性 | 知识条目按角色标记可见范围，检索时自动过滤 |
| 平台数据同步 | 对接校共体网站6大内容模块（示范校/新闻/政策/资源/活动/月报），定时增量同步 |

#### 4.2.3 案例脱敏与推荐

| 功能项 | 说明 |
|--------|------|
| 自动脱敏管道 | NER识别（人名/校名/地名）+ 正则匹配（联系方式/证件号）+ 规则替换 |
| 人工审核 | 脱敏结果须运营审核后方可入库，支持修改后通过 |
| 检索与展示分离 | 检索使用原始文本保证召回率，展示时实时替换为脱敏文本 |
| 智能推荐 | 问答后主动推荐相关成功实践案例（1-3条，卡片形式） |

#### 4.2.4 安全合规与内容治理

| 功能项 | 说明 |
|--------|------|
| 免责声明 | 每条AI回答末尾自动追加声明文案，运营可配置修改 |
| 双向敏感词过滤 | 输入侧：拒绝政治敏感/Prompt注入；输出侧：替换为安全回答 |
| 隐私信息检测 | 前端实时检测身份证号、手机号等，弹出提示建议用户删除 |
| 对话日志脱敏 | 存储前自动掩码敏感个人信息，运营后台展示脱敏版本 |
| 幻觉检测 | Prompt层约束 + 引用校验 + 条款编号正则校验 + 低置信度声明 |

#### 4.2.5 用户反馈与知识进化

| 功能项 | 说明 |
|--------|------|
| 满意度评分 | 1-5星评分，12px小尺寸不干扰阅读 |
| 不满意处理 | 运营直接回复，回复后自动标记完成，用户可查看处理结果 |
| 问答入库 | 反馈经运营回复后可"加入知识库"，向量化后即时生效 |
| 问题自动归集 | 用户新问题去重后进入待审核队列，运营批量审核 |

### 5.2 技术选型

| 层次 | 技术方案 | 说明 |
|------|---------|------|
| **前端** | Vue 3 + Element Plus + Pinia | 嵌入校共体网站，响应式适配桌面/移动端 |
| **后端框架** | JDK 21 + Spring Boot 3.x + Spring Cloud | 微服务架构，按业务拆分为2个服务 |
| **网关** | Spring Cloud Gateway | 统一路由、JWT鉴权、限流 |
| **服务通信** | Spring Cloud OpenFeign | 服务间同步调用 |
| **数据库** | PostgreSQL 16 + pgvector | 业务数据 + 向量数据统一管理 |
| **缓存** | Redis 7 | 会话上下文、语义缓存、分布式锁、Token黑名单 |
| **文档解析** | Apache PDFBox + Apache POI | PDF/Word文本、表格提取 |
| **认证** | JWT（access_token + refresh_token） | 手机号 + 验证码登录 |
| **数据库迁移** | Flyway | 版本化管理数据库Schema变更 |

### 5.3 AI模型方案

本项目不自建GPU算力，全部通过API调用方式对接大模型服务：

| 模型 | 参数量 | 用途 | 调用场景 |
|------|--------|------|---------|
| **Qwen3-32B** | 320亿 | 主推理引擎 | 问答生成、多轮对话、查询改写、NER脱敏、话题检测 |
| **Qwen3-VL-235B** | 2350亿 | 视觉理解 | 文档中表格/流程图的图像识别与文字描述 |
| **Qwen3-Embedding-8B** | 80亿 | 文本向量化 | 文档片段、用户查询、语义缓存的Embedding生成 |
| **Qwen3-Rerank-8B** | 80亿 | 检索重排序 | 混合检索结果精排，提升Top-K相关性 |

### 5.4 RAG检索引擎流程

```
用户提问
  ↓
敏感词过滤(输入) → 命中则拒绝
  ↓
语义缓存查询 → 命中则直接返回(相似度≥0.95, 按角色分组)
  ↓
查询改写(结合多轮上下文) → 独立完整查询
  ↓
混合检索: 向量Top-20 + 关键词Top-20 → RRF融合 → 角色可见性过滤 → Rerank取Top-5
  ↓
置信度评估 → 低于阈值则声明"暂无相关政策依据"
  ↓
LLM生成(Qwen3-32B, SSE流式) → 注入角色信息 + 强制引用 + 检索片段
  ↓
后处理: 敏感词过滤(输出) + 引用校验 + 免责声明追加 + 缓存写入
  ↓
返回用户
```

## 六、信息安全与合规保障

### 6.1 数据安全

| 安全措施 | 具体方案 |
|---------|---------|
| 传输加密 | 全链路HTTPS + HSTS |
| 数据存储 | 私有化部署，敏感信息加密存储 |
| 访问控制 | RBAC权限隔离（普通用户/运营人员），方法级鉴权 |
| 日志脱敏 | 用户消息入库前自动掩码身份证号/手机号/银行卡号 |
| 审计追踪 | 关键操作记录审计日志（登录/上传/删除/配置变更） |

### 6.2 内容合规

| 合规措施 | 具体方案 |
|---------|---------|
| AI免责声明 | 每条回答自动追加声明，文案运营可配置 |
| 双向敏感词 | 输入拦截 + 输出过滤，敏感词库运营维护 |
| 幻觉防控 | Prompt约束 + 引用后端校验 + 条款正则验证 + 低置信度声明 |
| 案例脱敏 | NER + 规则引擎 + 人工审核三层保障 |
| 隐私保护 | 前端实时隐私检测提醒，用户可选择修改后发送 |

## 八、部署与运维方案

### 8.1 部署架构

```
          用户浏览器
              │
              ▼
     ┌──────────────┐
     │   Nginx      │  SSL终止 + 静态文件 + 反向代理
     └──────┬───────┘
            │
            ▼
     ┌──────────────┐
     │   Gateway    │  路由 + 鉴权 + 限流
     │   (:8080)    │
     └───┬──────┬───┘
         │      │
         ▼      ▼
   ┌─────────┐ ┌─────────┐
   │edu-core │ │ edu-ai  │
   │ (:8081) │ │ (:8082) │
   └────┬────┘ └────┬────┘
        │           │
   ┌────┴────┐ ┌────┴────┐
   │   PG    │ │  Redis  │
   │ (:5432) │ │ (:6379) │
   └─────────┘ └─────────┘
```

### 8.2 硬件资源需求

| 资源 | 最低配置 | 推荐配置 | 说明 |
|------|---------|---------|------|
| 应用服务器 | 4核 8GB | 8核 16GB | 运行3个JVM进程（Gateway + edu-core + edu-ai） |
| 数据库服务器 | 2核 4GB, 50GB SSD | 4核 8GB, 100GB SSD | PostgreSQL + pgvector |
| 缓存服务器 | 1核 2GB | 2核 4GB | Redis 7 |
| 模型API | 外部调用 | 外部调用 | 无需GPU，通过API对接Qwen3系列 |

> 注：以上配置可部署在本地服务器或云主机，支持按需弹性扩展。

### 8.3 运维保障

| 运维项 | 方案 |
|--------|------|
| 健康检查 | Spring Boot Actuator `/actuator/health` |
| 日志管理 | 结构化JSON日志，按日轮转 |
| 数据备份 | PostgreSQL每日全量备份 + WAL归档 |
| 监控告警 | 服务可用性、API延迟、模型调用量、缓存命中率 |
| 系统可用性 | 目标≥99.5% |

## 十、预期建设成效

### 10.1 直接效益

| 效益维度 | 预期成效 |
|---------|---------|
| **服务效率提升** | 7×24小时在线服务，替代部分重复性政策咨询，预计减少50%以上常规咨询工作量 |
| **覆盖范围扩大** | 突破地域和时间限制，使乡村教师也能随时获得专业指导 |
| **政策传达加速** | 新政策上传后即可被检索和解读，信息传递周期从"周级"缩短为"分钟级" |
| **经验复用** | 优秀案例自动脱敏后入库，形成可检索的实践知识资产 |
| **决策支持** | 为管理者提供数据驱动的分析视角，辅助区域校共体建设决策 |

### 10.2 长期价值

- **知识资产沉淀**：通过持续运营积累形成校共体建设领域的专属知识库
- **服务能力进化**：用户反馈→知识入库闭环驱动系统能力持续提升
- **标准化推广**：成功经验可复制推广至其他省市的校共体或类似教育协作项目
- **数据洞察**：对话日志分析可揭示各地校共体建设的共性问题和薄弱环节
