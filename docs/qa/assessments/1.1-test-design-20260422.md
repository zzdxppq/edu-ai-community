## Scenarios by AC

### AC1: 后端项目初始化

> **场景**：空项目 → 执行初始化 → Spring Cloud + PG + Redis + Actuator 就绪
> **Dev 产物**：`backend/edu-gateway|edu-core|edu-ai|edu-common` 四模块 + `V1__init_extensions.sql` + `/api/health`

#### Core Scenarios

| ID | Lvl | Pri | Test | Why |
|----|-----|-----|------|-----|
| 1.1-INT-001 | I | P1 | `GatewayHealthEndpointTest`：`WebTestClient.get("/api/health")` 应返回 HTTP 200 + `"UP"` | AC1 main_scenario 直接断言；Actuator 挂载 `/api` 前缀的契约验证 |
| 1.1-INT-002 | I | P1 | `FlywayMigrationTest`（Testcontainers PG）：首次启动后 `SELECT extname FROM pg_extension WHERE extname IN ('vector','zhparser')` 返回两行 | BR-1.3 Flyway 迁移正确执行 `V1__init_extensions.sql`；后续 RAG Story 依赖 |
| 1.1-INT-003 | I | P2 | `FlywayIdempotencyTest`：连续两次启动 Flyway 不抛错，`flyway_schema_history` 表存在且 V1 标记 success | Flyway 基线幂等性 — 防止二次启动破坏环境 |
| 1.1-UNIT-001 | U | P1 | `RTest#success`：`R.success(data)` 生成 `{code:0, message:"OK", data:<payload>}` | `R<T>` 是所有 Controller 响应契约的基类，契约破坏影响全项目 |
| 1.1-UNIT-002 | U | P1 | `GlobalExceptionHandlerTest#handlesBizException`：抛出 `new BizException(4001,"x")` 被 `@RestControllerAdvice` 捕获，返回 `R{code:4001,message:"x",data:null}`，HTTP 200 | 异常契约基线 — 所有业务异常的统一出口 |
| 1.1-UNIT-003 | U | P2 | `GlobalExceptionHandlerTest#handlesValidation`：构造带 `@NotBlank` 字段缺失的请求，`MethodArgumentNotValidException` 映射为 `R{code:4000,...}` | Spring Validation 集成点验证 — 防止校验失败返回 Spring 默认 error JSON |
| 1.1-UNIT-004 | U | P2 | `GlobalExceptionHandlerTest#handlesUnknown`：抛出未知 `RuntimeException` → 返回 `R{code:5000,message:"系统繁忙",...}`，HTTP 500 | 兜底异常防止栈信息泄露（Low 安全风险） |

| 1.1-BLIND-BOUNDARY-001 | BOUNDARY | P2 | `RTest#successWithNullData`：`R.success(null)` 不抛 NPE，`data` 字段序列化为 `null`（非缺省） | BOUNDARY-001 Null Input |
| 1.1-BLIND-ERROR-001 | ERROR | P1 | `CoreApplicationStartupTest`（Testcontainers）：PG 容器停止 → edu-core 启动抛 `DataAccessResourceFailureException` → 日志 ERROR + 进程退出码非 0 | ERROR-002 Service Unavailable；AC1 Error Handling 契约 |

## Risk Coverage

本 Story 未生成 `risk-profile` 文档（Standard 级别未强制）。关键风险-场景映射：

| 隐含风险 | 覆盖场景 |
|---------|---------|
| R1：PG 启动连接失败未退出 → 假健康 | 1.1-BLIND-ERROR-001 |
| R2：Redis 不可达导致应用无法启动（过度降级 / 不降级均不妥） | 1.1-BLIND-ERROR-002 |
| R3：R\<T\> 契约破坏级联所有 Controller | 1.1-UNIT-001, 1.1-UNIT-002 |
| R4：pgvector 未启用，Story 2.x RAG 不可用 | 1.1-INT-002 |
| R5：前端网络错误泄露技术细节（违反 AC2） | 1.1-BLIND-ERROR-004 |
| R6：Redis 序列化不统一导致二期解码失败 | 1.1-BLIND-RESOURCE-001 |

## Execution Order

1. **P1 Unit**（快速失败层）：1.1-UNIT-001 → 002 → 005 → 006
2. **P1 Integration**（骨架验证）：1.1-INT-001 → 1.1-INT-002 → 1.1-BLIND-ERROR-001 → 1.1-BLIND-ERROR-004
3. **P2 Unit**：1.1-UNIT-003/004/007/008/009 + 1.1-BLIND-BOUNDARY-001/002 + 1.1-BLIND-FLOW-001
4. **P2 Integration**：1.1-INT-003 + 1.1-BLIND-ERROR-002/003 + 1.1-BLIND-RESOURCE-001

## Coverage Gaps

无 — 每个 AC 至少 1 个 P1 场景；所有 Error Handling 列项均有对应场景；pgvector/Flyway/Redis 等架构约束均覆盖。E2E 故意留空，由 Epic 1 收尾 smoke test 统一验证（符合 Dev Notes "禁止事项"）。
