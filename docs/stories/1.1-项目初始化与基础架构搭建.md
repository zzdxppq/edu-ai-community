## Story

**As a** 开发团队,
**I want** 搭建前后端项目的基础骨架（Spring Cloud 后端、Vue 3 前端、PostgreSQL + Redis 基础设施）,
**so that** 后续所有业务 Story 能在稳定、统一的工程底座上并行开发。

---

## Epic Context

**Epic 1 - 基础设施与用户系统**

建立项目基础设施（前后端框架、数据库、缓存），实现完整的用户系统，包括手机号注册/登录、角色选择、JWT 认证保持登录、使用日志记录。此 Epic 是所有后续功能的基础依赖。

**Story 1.1 Deliverables:**

- Spring Cloud 微服务后端项目骨架（edu-common / edu-gateway / edu-core / edu-ai 四模块）
- PostgreSQL 16 + pgvector + zhparser 连接与扩展启用
- Redis 7 连接与缓存/序列化配置
- Flyway 迁移机制初始化
- 健康检查接口 `GET /api/health`（Spring Boot Actuator）
- Vue 3 + Element Plus 前端项目骨架（Vite + TS + Pinia + Vue Router + Axios）
- 前端响应式布局基线（CSS 变量、断点定义、首页占位）
- 本地开发环境一键启动脚本/文档

## Acceptance Criteria

### AC1: 后端项目初始化

**Scenario**
```gherkin
GIVEN 空项目仓库
WHEN 开发人员执行项目初始化
THEN
  - Spring Boot 3.x + Spring Cloud 微服务项目结构创建完成，包含 Controller、Service、Repository 分层目录
  - PostgreSQL 数据库连接配置完成（Spring Data JPA）
  - Redis 7 连接配置完成（Spring Data Redis）
  - 健康检查接口 GET /api/health 返回 200（Spring Boot Actuator）
```

**Business Rules**
| ID | Rule |
|----|------|
| BR-1.1 | 项目采用 JDK 21 + Spring Boot 3.x + Spring Cloud 微服务框架，按业务拆分 1-2 个服务模块 |
| BR-1.2 | 数据库使用 PostgreSQL + pgvector 扩展 |
| BR-1.3 | 使用 Flyway 进行数据库迁移管理 |

**Error Handling**
| Scenario | Code | Message | Action |
|----------|------|---------|--------|
| 数据库连接失败 | 503 | 服务暂时不可用 | 启动时检查连接，失败则记录日志并退出 |

### Database Design

本 Story **不创建任何业务表**。仅完成以下 DDL：

- **扩展启用**（通过 Flyway `V1__init_extensions.sql`，部署在 `edu-ai` 模块）：
  - `CREATE EXTENSION IF NOT EXISTS vector;`
  - `CREATE EXTENSION IF NOT EXISTS zhparser;`
  - `CREATE TEXT SEARCH CONFIGURATION zhcfg (PARSER = zhparser);`（若不存在）
- **Flyway 基线表** `flyway_schema_history`：由 Flyway 自动创建，每个服务模块独立维护迁移路径（`edu-core` 与 `edu-ai` 各自一份）
- 全部建表 DDL 延后到 Story 1.2（`users`）、Story 1.3（会话 token 无新表）、Story 1.4（`conversations`、`messages`）等后续 Story 执行 `[Source: docs/fullstack-architecture/database-schema.md]`

### Data Synchronization Requirements

### Data Models

本 Story 仅交付基础模型骨架，**不定义具体业务 DTO/Entity**：

- `edu-common` 的 `GlobalExceptionHandler`（`@RestControllerAdvice`），统一捕获 `BizException` / `MethodArgumentNotValidException` / `Exception` 并返回 `R<T>` 结构

- **`CREATE EXTENSION` 生产权限假设未声明** (Tasks T4 / Dev Notes → Database Design): Flyway 在 `edu-ai` 启动时执行 `CREATE EXTENSION vector; CREATE EXTENSION zhparser;` 需要数据库用户具备 SUPERUSER 权限，生产环境普通应用用户通常不具备此权限。本地开发（容器内 `postgres` 超级用户）可直接执行，约束已覆盖本 Story 验证范围，但长远（部署阶段）需由 DBA 预执行。
  - 建议：在 Dev Notes → Technical Constraints 追加一行「生产部署：`CREATE EXTENSION` 由 DBA 预执行；Flyway 迁移保留 `IF NOT EXISTS` 以保持幂等」。

- **edu-common 被依赖关系未显式声明** (Tasks T1 / deliverable_bindings): 系统总览 §1.4 标注 edu-common 为"被其他模块依赖的公共模块"，但 Story T1 子任务仅声明创建四模块骨架，未明确要求 `edu-gateway` / `edu-core` / `edu-ai` 的 `pom.xml` 声明对 `edu-common` 的 Maven 依赖。Dev 可能遗漏导致 `R<T>` / `BizException` / `GlobalExceptionHandler` 无法在其他模块中 import。
  - 建议：在 T1 补充子任务「`edu-gateway` / `edu-core` / `edu-ai` 的 `pom.xml` 声明 `<dependency>com.edu.ai:edu-common</dependency>` 以启用 R<T> 与 GlobalExceptionHandler」。

- **Gateway 健康聚合语义过度承诺** (Tasks T5 第4条子任务): "edu-core 与 edu-ai 分别暴露 `/actuator/health` 供 Gateway 健康聚合" 暗示 Gateway 会聚合下游健康状态，但 Spring Boot Actuator 默认不会自动聚合下游微服务健康，需自定义 `HealthIndicator` 或使用 Spring Cloud 的 gateway health contributor。而 AC1 只要求 `GET /api/health` 返回 200 + `"UP"`（即 Gateway 自身），聚合非必需。
  - 建议：将该子任务语义明确为「可选：edu-core/edu-ai 暴露 `/actuator/health`，Gateway 健康聚合实现延后到可观测性 Story（见 docs/fullstack-architecture/15-可观测性.md）」，避免 Dev 误实现自定义 HealthIndicator 造成范围蔓延。
